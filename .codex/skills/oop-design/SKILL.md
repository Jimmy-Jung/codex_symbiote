# OOP Design

## 목표
객체의 역할/책임/협력을 명확히 하고, 정적/동적 설계를 합의한 뒤 구현으로 전환한다.

## 워크플로우
1. 기준 확보
- Context7으로 관련 언어/프레임워크 문서를 먼저 확인한다.
- 코드베이스에서 유사 패턴(레이어, 네이밍, DI 방식)을 수집한다.

2. 요구사항 정리
- `What`, `Why`, `How`, `Success Criteria`를 짧게 정리한다.
- 누락된 핵심 질문 3~7개를 도출한다.

3. 정적 설계
- 후보 객체를 Layer별로 정리한다.
- 각 객체의 Role, Responsibility, Collaboration을 정의한다.
- Mermaid `classDiagram`으로 구조를 확정한다.

4. 동적 설계
- 핵심 시나리오 1~2개를 선택한다.
- Mermaid `flowchart`로 메시지 흐름을 작성한다.

5. 리스크 리뷰
- DIP/OCP 위반, 책임 누수, 과한 결합, 트랜잭션/동시성 경계를 점검한다.
- 이슈를 `높음/중간/낮음`으로 분류하고 근거를 남긴다.

6. 구현 전환
- 사용자가 `설계 완료` 또는 `설계 완료, 구현 시작`을 명시한 경우에만 구현한다.
- 그 전에는 코드/파일 변경 없이 설계 산출물만 다룬다.

7. 검증
- 설계 산출물 간 정합성을 점검한다.
- 구현 단계라면 프로젝트 표준 검증(빌드/테스트/린트)을 실행하고 결과를 요약한다.

## 멀티/서브 에이전트 운영
기본은 단일 에이전트로 진행한다. 아래 조건에서만 선택적으로 멀티/서브 에이전트를 사용한다.

사용 조건:
- 코드베이스 탐색 범위가 크고, 읽기 작업 병렬화가 유리할 때
- 설계안 비교/리스크 리뷰를 독립적으로 교차 검증할 때

권장 분담:
- `explorer`: 유사 코드/의존성/경계 탐색 (read-only)
- `architect`: 정적 구조 대안 설계
- `reviewer` 또는 `critic`: DIP/OCP/회귀 리스크 검토

운영 규칙:
- 병렬화는 읽기/분석 작업에 우선 적용한다.
- 쓰기(코드 수정)는 충돌 방지를 위해 단일 주체가 최종 반영한다.
- 서브 에이전트는 원본 로그 대신 요약 결과만 반환한다.
- `multi_agent` 기능이 꺼져 있으면 단일 에이전트로 폴백하고 사유를 보고한다.

## 설계 규칙
- 구현 방법보다 객체가 제공하는 기능(무엇)을 먼저 정의한다.
- 절차형 분기보다 메시지 기반 협력을 우선한다.
- 레이어 기본 원칙: `Interface -> Application -> Domain -> Infrastructure`.
- 특정 언어 문법에 과적합된 설계를 강제하지 않는다.
- 성능보다 가독성/변경 용이성을 우선한다.
- 존재하지 않는 타입/모듈/API를 가정하지 않는다.

## 응답 포맷
1. 설계 요청: 변경/결정 요약 -> 산출물 -> 열린 이슈 -> 다음 액션
2. 리뷰 요청: 심각도 순 이슈 -> 파일/라인 근거 -> 권장 수정
3. 구현 요청: 변경 요약 -> 파일 경로/라인 -> 검증 결과
4. 불확실한 사실은 추측하지 말고 확인 필요 항목으로 분리
